# Integrate Azure OpenAI & Bright Data

## What & Why
Wire up the Azure OpenAI and Bright Data credentials that have been provided so the app delivers real AI-generated answers instead of the current hardcoded Hindi template. Bright Data SERP supplements the local scheme dataset with live web results. Both paths must degrade gracefully to the offline keyword+template fallback when credentials are absent or APIs fail.

## Done looks like
- Querying the app (e.g. "pension schemes") returns a fluent, contextual answer generated by Azure OpenAI GPT instead of the template string
- The answer is enriched with live web snippet context from Bright Data SERP when the credentials are present
- If Azure OpenAI or Bright Data is unavailable (or `OFFLINE_ONLY=true`), the app falls back cleanly to the existing keyword search + template answer with no crash
- All env vars are persisted in `.env` (gitignored) and documented as empty placeholders in `.env.example`
- `requirements.txt` (root) includes the `openai` SDK; `backend/requirements.txt` mirrors it via `-r ../requirements.txt`
- `cd backend && pytest` passes without regressions

## Out of scope
- Replacing the translation service (still uses `deep-translator`)
- FAISS / sentence-transformers vector search (disk quota constraint; keyword search remains)
- Any frontend changes
- Azure Speech Services (browser speech API stays)

## Tasks
1. **Secrets & config** — Store all provided env vars in `.env`, update `.env.example` with blank placeholders, and extend `backend/app/config.py` with typed fields for `OFFLINE_ONLY`, `AZURE_OPENAI_*` (endpoint, key, deployment, API version), and `BRIGHTDATA_*` (API token, SERP zone, DC proxy host/port/user/pass).

2. **Install openai SDK** — Add `openai` to the root `requirements.txt` and install it; confirm `backend/requirements.txt` references root via `-r ../requirements.txt`.

3. **Azure OpenAI answer generation** — Update `RAGEngine.generate_answer()` to call Azure OpenAI chat completions (using the `openai` SDK's `AzureOpenAI` client) with the retrieved scheme context and the user's English question. Wrap the call in a try/except so any API error (or `OFFLINE_ONLY=true` or missing credentials) falls back to the existing template string.

4. **Bright Data SERP service** — Create `backend/app/services/brightdata_service.py` with a `search_schemes(query)` function that POSTs to the Bright Data SERP API using the configured token and zone, returns the top 3 organic snippet texts, and returns an empty list silently on any failure or when credentials are absent.

5. **Wire Bright Data into query pipeline** — In `backend/app/api/query.py`, call the Bright Data service after keyword search and prepend any returned snippets to the context string passed to `generate_answer`, so the LLM has both local and live knowledge.

6. **Verify** — Run `cd backend && pytest` and smoke-test a live query end-to-end to confirm the answer quality improves and fallback still works.
